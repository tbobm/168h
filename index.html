Porteur de projet

L'entreprise qui propose ce défi est Enix.

Pitch de Enix :

Notre solution permet le multiplexage et l'affichage de logs sur une interface web moderne.

Le référent technique qui vous suivra sur ce code camp est Arthur CROS.

Consignes

Contexte :

Lorsqu'on lance une app sur plusieurs machines distinctes (via Kubernetes ou non), on a rapidement envie d'avoir des logs consolidés.
Sans faire appel à des services en Saas, ElasticSearch/Kibana et Graylog qui sont les solutions habituelles, les machines sont malheureusement très lourdes en termes de mise en place et en ressources.

L'idéal serait d'avoir l'équivalent de la commande shell tail sur l'ensemble des logs, sans chercher à faire de l'indexation de log.

C'est ce que l'on propose de construire dans ce projet (un peu à l'image de papertrail.com) : pas de persistence, léger et sans configuration, avec une vue instantanée des nouveaux logs et avec en option la possibilité de filtrer par host ou par conteneur.

L'idée est de récupérer les logs par les outils habituels : fluentd, filebeat, ou fluentbeat.
De les centraliser (avec logstash), en option sans.
Et de profiter des websockets pour afficher dans un webui moderne les logs en live.

Votre défi :

Implémenter une solution plus simple et plus élégante que la plupart des outils utilisés par l'industrie aujourd'hui.

Le livrable :

Un repo git, un conteneur et une documentation permettant d'utiliser le projet en moins de 5 minutes.

Spécifications technico-fonctionnelles

Technologies préconisées :

  • conteneurs
  • websocket
  • js
  • compatibilité kubernetes
Certains se poseront peut-être ces questions :

Est-il obligatoire d'utiliser les technologies proposées (fluentd, filebeat, fluentbeat, logstash) ou sommes-nous libre d'utiliser les technologies que nous voulons ?
Quel genre de container devons-nous rendre ?
Un container simple avec logstash (sans web app containerisée par exemple ...) ?

Le but est de vous laisser un maximum de liberté tant que le résultat est là, mais pour compléter :

  • le but principal est d'avoir une interface web qui permet de consommer les logs venant de multiples hosts ou conteneurs ou autre à l'image de ce que fait https://www.papertrail.com/

  • le but n'est pas d'être iso fonctionnalités (cela est pour l'ui)

  • Concernant les internals : websocket ou équivalent est un must afin d'avoir un minimum de latence entre l'arrivée des logs et leur affichage

  • Concernant la source des logs, dans la majorité des cas de production actuels, vous trouverez l'un des trois outils présents dans le sujet.
    Dans la mesure où Enix veut pouvoir installer le projet le plus facilement possible, il est déconseillé d'implémenter un truc similaire à ce qui existe déjà. Par contre, si vous trouvez un outil équivalent à ceux listés, vous pouvez le proposer à Arthur CROS.
